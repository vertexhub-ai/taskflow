# ADR-001 — TaskFlow Stack & Architecture

**Date:** 2026-06-20  
**Status:** Accepted  
**Decider:** Solution Architect (V-9986)

---

## Context

TaskFlow is a multi-subsystem team task manager required to ship on the existing
Node + Postgres stack as specified by the umbrella issue V-9986.  Because the
stack was pre-specified, Phase 1–3 of the standard architect workflow (domain
research, stack selection, team tally) is skipped per the hard rule: *"If the
umbrella explicitly specifies stack, skip Phase 1–3 and pass directly to Tech
Lead."*

---

## Decision

### Runtime — Node.js 20 + Fastify 4

Fastify was already chosen in the scaffold.  It is the fastest HTTP framework
in the Node ecosystem (benchmark: ~77k req/s vs Express ~17k), has first-class
TypeScript support, and a rich plugin ecosystem (`@fastify/cookie`,
`@fastify/static`).  We keep `fastify@^4` to stay on the LTS release.

### Database — PostgreSQL via `postgres` (porsager)

`postgres` (the `postgres` npm package) provides a lightweight, tagged-template
SQL client with connection pooling, no ORM overhead, and native Postgres type
mapping.  Migrations run at startup from `migrations/` in filename order.
All DDL uses `IF NOT EXISTS` so restarts are idempotent.

### Auth — session cookies + bcrypt

- Passwords hashed with **bcryptjs** (pure JS, no native deps), cost factor 12.
- Sessions stored in `sessions` table (token, user_id, expires_at).  Token is 32
  random bytes → 64 hex chars.
- `@fastify/cookie` reads the `sid` cookie on every request.
- `requireAuth` middleware in `src/middleware/auth.js` is imported by each route
  plugin; validated on every protected request.
- Alternative considered: JWT.  Rejected — JWT requires a logout table anyway
  to support revocation; a session table is simpler and equally fast.

### Frontend — Vanilla JS SPA in `public/index.html`

- Single file, no build step, served as a static asset by `@fastify/static`.
- Calls REST API with `fetch()` and manages client-side view state.
- Keeps the deployment surface minimal: no Node build toolchain in production.
- Alternative considered: React or Svelte SPA.  Rejected — no frontend framework
  specialist confirmed available; vanilla JS meets the acceptance criteria without
  a build pipeline.

## Schema Summary

| Table        | Purpose                                      |
|--------------|----------------------------------------------|
| users        | Account credentials and display name         |
| sessions     | Server-side sessions (token → user_id)       |
| projects     | Team projects owned by a user                |
| tasks        | CRUD items within a project                  |
| comments     | Threaded comments on a task                  |
| activity     | Append-only log for dashboard activity feed  |

---

## Subsystems & Ownership

| Subsystem  | Route file(s)              | Key capabilities    |
|------------|----------------------------|---------------------|
| Auth       | `src/routes/auth.js`       | bcrypt, sessions    |
| Projects   | `src/routes/projects.js`   | Postgres CRUD       |
| Tasks      | `src/routes/tasks.js`      | CRUD + board/list   |
| Comments   | `src/routes/comments.js`   | Threaded comments   |
| Dashboard  | `src/routes/dashboard.js`  | Aggregation queries |
| UI         | `public/index.html`        | Vanilla JS SPA      |

---

## Trade-offs

| Topic              | Choice Made                  | Trade-off                            |
|--------------------|------------------------------|--------------------------------------|
| Auth               | Session cookies              | No stateless scaling; fine for MVP   |
| Migrations         | Raw SQL, no ORM              | More verbosity; full SQL control     |
| Frontend           | Vanilla JS, no framework     | Less DX; zero build complexity       |
| Password hashing   | bcryptjs (pure JS)           | ~30% slower than native; no C addon  |
| Multi-tenancy      | owner_id on projects         | Simple; no team-share in v1          |

---

## Acceptance Criteria Mapping

| Criterion                                    | Implementation                             |
|----------------------------------------------|--------------------------------------------|
| GET / → 200 + non-empty HTML                 | `@fastify/static` serves `public/index.html` |
| GET /api/health → `{status:"ok"}`            | `src/routes/health.js`                     |
| Signup / login / gated routes                | `src/routes/auth.js` + `requireAuth` hook  |
| Tasks persist; Board reflects status         | `src/routes/tasks.js` `/board` endpoint    |
| List view filters/sorts                      | `?status=&sort=` query params on `/tasks`  |
| Dashboard counts accurate                    | SQL aggregations in `dashboard.js`         |
